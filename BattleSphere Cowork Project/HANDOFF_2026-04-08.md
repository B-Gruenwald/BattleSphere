# BattleSphere – Session Handoff (2026-04-08)

## Project overview
BattleSphere is a Next.js (App Router) web app for running map-based Warhammer 40K campaigns. Stack: React 19, Tailwind CSS v4, Supabase (Postgres + RLS + Storage). Live on Vercel at https://battle-sphere-topaz.vercel.app. GitHub: https://github.com/B-Gruenwald/BattleSphere.

## File access — do this first
The source code is NOT in the default workspace folder. At the start of every session you must request access to the source directory:

```
request_cowork_directory: ~/Desktop/BattleSphere
```

This mounts to `/sessions/<session-id>/mnt/BattleSphere`. All code edits go here.
The "BattleSphere Cowork Project" folder (SQL migrations, handoffs) is a separate mount — it is already available at `/sessions/<session-id>/mnt/BattleSphere Cowork Project`.

## Autonomous git push — how to do it
The git remote already has the token embedded, so `git push` works directly. Typical flow:

```bash
cd /sessions/<session-id>/mnt/BattleSphere
git add <files>
git commit -m "..."
git push
```

If commit fails with `HEAD.lock` file exists error:
1. Call `mcp__cowork__allow_cowork_file_delete` for the lock file path
2. `rm -f .git/HEAD.lock`
3. Re-run the commit

Vercel auto-deploys on every push to `main`.

### Technical notes for Supabase Storage
- Use `createClient()` (client-side) for uploads from the browser.
- Upload: `supabase.storage.from('territory-images').upload(path, file, { upsert: true })`
- Get public URL: `supabase.storage.from('territory-images').getPublicUrl(path).data.publicUrl`
- The bucket must exist and have a public policy before the app can use it. Provide a setup checklist for Benjamin to run in the Supabase dashboard.
- File size: recommend a 5 MB client-side limit. Accept jpg/png/webp.

### Supabase Storage bucket setup (Benjamin runs this manually)
In Supabase dashboard → Storage → New bucket:
- Name: `territory-images`
- Public: ✓ (toggle on)
Then in SQL editor:
```sql
CREATE POLICY "Public read access"
  ON storage.objects FOR SELECT
  USING (bucket_id = 'territory-images');

CREATE POLICY "Authenticated upload"
  ON storage.objects FOR INSERT
  WITH CHECK (bucket_id = 'territory-images' AND auth.role() = 'authenticated');

CREATE POLICY "Authenticated update/delete"
  ON storage.objects FOR DELETE
  USING (bucket_id = 'territory-images' AND auth.role() = 'authenticated');
```

## Key technical reminders
- All files use `.js` — never `.tsx` or TypeScript syntax
- Always use `select('*')` for Supabase queries
- `'use client'` required for any component with event handlers, useState, file inputs
- Server components (page.js files) use `createClient` from `@/lib/supabase/server`
- Client components use `createClient` from `@/lib/supabase/client`
- Next.js middleware is `proxy.js` exporting `proxy` (not `middleware`)
- Benjamin is non-technical — keep explanations clear and non-technical
